# gazillion-lotp

Disposable LOTP shooting range for SmokedMeat.

## Purpose

This repository exists to battle test LOTP coverage without touching a real product repository.

The goals are:
- confirm which LOTP families are detected by stock `poutine`
- confirm which families SmokedMeat can actually exploit today
- give each family a small, isolated GitHub Actions workflow for manual exploratory testing

This is not meant to be a pure copy of `lotp-sandbox`, and it is not meant to be a CTF. The repo is intentionally vulnerable and optimized for quick validation loops.

## Workflow Model

Each workflow is intentionally tiny and follows the same unsafe pattern:

1. trigger on `pull_request_target`
2. use `paths:` to limit execution to one relevant LOTP slice
3. `actions/checkout` the attacker-controlled `github.event.pull_request.head.sha`
4. run one tool command that should exercise the target LOTP family

The `paths:` gates are the main control knob. A PR that only touches `terraform/**` should only run the Terraform workflow. A PR that only touches `package.json` should only run the NPM workflow. That keeps manual testing focused and avoids lighting up the whole repo at once.

## Repository Layout

The repo uses a hybrid layout.

- Root-level fixtures are reserved for the SmokedMeat payload paths that are supported today.
- Per-tool directories mirror the `lotp-sandbox` style for detection-only families and for families that need room for multiple files or notes.
- Each workflow should stay scoped to one tool family or one narrow variant.

This split exists because SmokedMeat currently generates several payloads to fixed root paths such as `package.json`, `.yarnrc.yml`, `setup.py`, `build.rs`, and `Makefile`. A pure directory-only layout would be cleaner, but it would break path-gated validation for the subset that SmokedMeat can already deploy.

## SmokedMeat-Supported Today

The current auto-exploit subset is:

- `bash`
- `powershell`
- `python`
- `npm`
- `yarn`
- `pip`
- `cargo`
- `make`

Those families keep their fixtures on the exact paths SmokedMeat uses today so a PR generated by SmokedMeat can trigger the matching workflow directly.

## Detection Coverage

The current repo shape contains:

- 31 tiny GitHub Actions workflows
- 30 unique LOTP tool families detected by stock `poutine` `untrusted_checkout_exec`
- 1 extra duplicate `pip` finding because the `pip` workflow trips two matching commands

Detection-only families are still useful here even when SmokedMeat cannot deploy them yet. They let us confirm that stock `poutine` sees them and give us a place to test future SmokedMeat support incrementally.

## Adding More Variants

When adding a new family or variant:

- if SmokedMeat can generate it today, prefer keeping the trigger path compatible with the current payload generator
- if SmokedMeat cannot generate it today, prefer a dedicated tool directory and gate the workflow on that directory
- keep workflows tiny and avoid mixing unrelated LOTP families into the same file
- avoid shared trigger paths across different families unless the collision is intentional

## Manual Testing Loop

Typical manual validation flow:

1. open or update a PR that only touches the files for one LOTP family
2. verify that only the expected tiny workflow runs
3. confirm stock `poutine` detects the workflow shape
4. confirm whether SmokedMeat can land a working payload for that family

The most important question for this repo is not "is this realistic production code?" but "does this family detect cleanly, and can SmokedMeat drive it end-to-end yet?"
